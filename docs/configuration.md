# GoGoClaw Configuration Reference

## Overview

All GoGoClaw configuration lives under `~/.gogoclaw/`. The config system uses a multi-file YAML structure where each concern (providers, agents, channels, network, memory, MCP servers) is defined in its own file or directory.

Environment variables can be referenced in any YAML value using the `${VAR_NAME}` syntax. Unset variables resolve to empty strings.

On first run, the bootstrap process auto-generates most configuration files through an interactive setup conversation. The bootstrap creates the directory structure, copies default templates, and writes provider, agent, channel, and network configs based on your answers. An `initialized` marker file is created at `~/.gogoclaw/initialized` once bootstrap completes successfully.

### Directory layout

```
~/.gogoclaw/
  config.yaml            # Core config (workspace, logging, storage)
  network.yaml           # Network allowlist
  env                    # Environment variables (KEY=value)
  initialized            # Bootstrap completion marker
  providers/             # One YAML per LLM provider
    openai.yaml
    ollama.yaml
  agents/                # One YAML per agent profile
    base.yaml
    base.md              # System prompt for the base agent
    user.md              # User profile (generated by bootstrap)
  channels/              # One YAML per communication channel
    rest.yaml
    telegram.yaml
  mcp/                   # One YAML per MCP server
    filesystem.yaml
  memory/
    config.yaml          # Memory system config
    daily/               # Daily memory storage
  workspace/
    inbox/
    outbox/
    scratch/
    documents/
  skills.d/              # WASM skill bundles
  audit/
    gogoclaw.jsonl        # Structured audit log
  data/
    conversations.db      # SQLite conversation store
```

## Core Config (config.yaml)

The top-level `config.yaml` controls workspace paths, logging, and storage.

```yaml
workspace:
  base: "~/.gogoclaw/workspace"
  inbox: "~/.gogoclaw/workspace/inbox"
  outbox: "~/.gogoclaw/workspace/outbox"
  scratch: "~/.gogoclaw/workspace/scratch"
  documents: "~/.gogoclaw/workspace/documents"

logging:
  level: "info"    # debug | info | warn | error
  audit:
    enabled: true
    path: "~/.gogoclaw/audit/gogoclaw.jsonl"
    # At-rest encryption using AES-256-GCM. Requires passphrase env var or auto-generated key.
    encrypt: false

storage:
  conversations:
    path: "~/.gogoclaw/data/conversations.db"
    # At-rest encryption using AES-256-GCM. Requires passphrase env var or auto-generated key.
    encrypt: false
    passphrase_env: "GOGOCLAW_DB_PASSPHRASE"
  ephemeral_mode: false
```

### Fields

| Field | Type | Default | Description |
|---|---|---|---|
| `workspace.base` | string | `~/.gogoclaw/workspace` | Root workspace directory |
| `workspace.inbox` | string | `~/.gogoclaw/workspace/inbox` | Incoming file drop |
| `workspace.outbox` | string | `~/.gogoclaw/workspace/outbox` | Outgoing file pickup |
| `workspace.scratch` | string | `~/.gogoclaw/workspace/scratch` | Temporary working area |
| `workspace.documents` | string | `~/.gogoclaw/workspace/documents` | Persistent documents |
| `logging.level` | string | `info` | Log level: `debug`, `info`, `warn`, `error` |
| `logging.audit.enabled` | bool | `true` | Enable structured audit logging |
| `logging.audit.path` | string | `~/.gogoclaw/audit/gogoclaw.jsonl` | Audit log file path |
| `logging.audit.encrypt` | bool | `false` | Encrypt audit log entries at rest (enc:v1: format) |
| `storage.conversations.path` | string | `~/.gogoclaw/data/conversations.db` | SQLite database path |
| `storage.conversations.encrypt` | bool | `false` | Encrypt message content and tool_calls at rest |
| `storage.conversations.passphrase_env` | string | `GOGOCLAW_DB_PASSPHRASE` | Env var containing passphrase for key derivation (Argon2id) |
| `storage.ephemeral_mode` | bool | `false` | When true, conversations are not persisted |

## Provider Configs (providers/*.yaml)

Each file in `~/.gogoclaw/providers/` defines an LLM provider connection. The filename (without `.yaml`) becomes the provider key used in agent configs.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | No | Display name (defaults to filename) |
| `type` | string | Yes | `openai_compatible` or `ollama` |
| `base_url` | string | Yes | API base URL |
| `api_key` | string | No | API key (use `${ENV_VAR}` to reference env vars) |
| `default_model` | string | No | Model identifier |
| `max_context_tokens` | int | No | Maximum context window size |
| `timeout` | duration | No | Request timeout (e.g. `60s`, `2m`) |
| `retry` | int | No | Number of retry attempts on failure |

### Example: OpenAI (providers/openai.yaml)

```yaml
name: "openai"
type: "openai_compatible"
base_url: "https://api.openai.com/v1"
api_key: ${OPENAI_API_KEY}
default_model: "gpt-4o-mini"
max_context_tokens: 128000
timeout: 60s
retry: 1
```

### Example: Ollama (providers/ollama.yaml)

```yaml
name: "ollama"
type: "ollama"
base_url: "http://localhost:11434/v1"
default_model: "llama3"
max_context_tokens: 8192
timeout: 120s
retry: 0
```

### Example: Groq (providers/groq.yaml)

```yaml
name: "groq"
type: "openai_compatible"
base_url: "https://api.groq.com/openai/v1"
api_key: ${GROQ_API_KEY}
default_model: "llama-3.3-70b-versatile"
max_context_tokens: 131072
timeout: 30s
retry: 1
```

### Example: MiniMax (providers/minimax.yaml)

```yaml
name: "minimax"
type: "openai_compatible"
base_url: "https://api.minimax.chat/v1"
api_key: ${MINIMAX_API_KEY}
default_model: "MiniMax-Text-01"
max_context_tokens: 1000000
timeout: 60s
retry: 1
```

## Agent Configs (agents/*.yaml)

Each file in `~/.gogoclaw/agents/` defines an agent profile. The `base.yaml` agent is loaded by default.

### Fields

| Field | Type | Default | Description |
|---|---|---|---|
| `name` | string | filename | Display name |
| `inherits` | string | -- | Name of another agent config to inherit from |
| `system_prompt_file` | string | -- | Filename of system prompt (relative to agents/) |
| `provider_routing.mode` | string | -- | Routing mode (see below) |
| `provider_routing.provider_chain` | list | -- | Ordered list of providers to try |
| `pii.mode` | string | `disabled` | PII detection mode (see PII Modes section) |
| `skills.always_available` | bool | `true` | Make all skills available without explicit listing |
| `skills.auto_discover` | bool | `true` | Automatically discover skills in skills.d/ |
| `skills.allowed` | list | -- | Explicit list of allowed skill names |
| `context.max_history_tokens` | int | `4096` | Max tokens of history to include in context |
| `context.summarization.enabled` | bool | `true` | Enable rolling summarization |
| `context.summarization.provider` | string | -- | Provider to use for summarization (optional) |
| `context.summarization.threshold_tokens` | int | `3072` | Token count that triggers summarization |
| `memory.enabled` | bool | `true` | Enable vector memory retrieval |
| `memory.top_k` | int | `10` | Number of memories to retrieve |
| `memory.relevance_threshold` | float | `0.7` | Minimum cosine similarity score |
| `memory.recency_weight` | float | `0.2` | Weight given to recency vs relevance (0.0-1.0) |
| `shell.confirmation` | string | `always` | Shell confirmation mode: `always`, `destructive_only`, `never` |
| `network.additional_allowlist` | list | -- | Extra domains to allow for this agent |

### Provider Routing Modes

- **`cloud-only`** -- All requests go to cloud providers. Use when you have a reliable API key and no local LLM.
- **`local-only`** -- All requests go to local providers (Ollama). Use for full offline operation or maximum privacy.
- **`hybrid`** -- Try local provider first, fall back to cloud if local is unavailable or fails. Good for development where you want speed when Ollama is running but a fallback otherwise.
- **`tiered`** -- Reserved for future use. Will route requests to different providers based on task complexity and data classification, enabling architectures like: fast local 9B for triage, local 27B for daily agent work, local 120B for privacy-sensitive complex tasks, and cloud API for complex non-sensitive tasks.

### Example: agents/base.yaml

```yaml
name: "GoGoClaw Assistant"
system_prompt_file: "base.md"

provider_routing:
  mode: "cloud-only"
  provider_chain:
    - provider: "openai"
      timeout: 60s
      retry: 1

pii:
  mode: "disabled"

skills:
  always_available: true
  auto_discover: true

context:
  max_history_tokens: 4096
  summarization:
    enabled: true
    threshold_tokens: 3072

memory:
  enabled: true
  top_k: 10
  relevance_threshold: 0.7
  recency_weight: 0.2

shell:
  confirmation: "always"
```

### Example: Hybrid agent with local preference

```yaml
name: "Private Assistant"
inherits: "base"
system_prompt_file: "base.md"

provider_routing:
  mode: "hybrid"
  provider_chain:
    - provider: "ollama"
      timeout: 120s
      retry: 0
    - provider: "openai"
      timeout: 60s
      retry: 1

pii:
  mode: "strict"

shell:
  confirmation: "destructive_only"

network:
  additional_allowlist:
    - "internal.example.com"
```

## Channel Configs (channels/*.yaml)

Each file in `~/.gogoclaw/channels/` defines a communication channel.

### REST Channel (channels/rest.yaml)

| Field | Type | Default | Description |
|---|---|---|---|
| `name` | string | `rest` | Display name |
| `enabled` | bool | `false` | Enable the REST API server |
| `listen` | string | `127.0.0.1:8080` | Address and port to bind |
| `api_key_env` | string | `GOGOCLAW_REST_API_KEY` | Env var name containing the API key for authentication |
| `allowed_origins` | list | -- | CORS allowed origins |

```yaml
name: "rest"
enabled: false
listen: "127.0.0.1:8080"
api_key_env: "GOGOCLAW_REST_API_KEY"
allowed_origins:
  - "http://localhost:3000"
```

### Telegram Channel (channels/telegram.yaml)

| Field | Type | Default | Description |
|---|---|---|---|
| `name` | string | `telegram` | Display name |
| `enabled` | bool | `false` | Enable the Telegram bot |
| `token_env` | string | `GOGOCLAW_TELEGRAM_TOKEN` | Env var name containing the bot token |
| `allowed_users` | list | `[]` | Usernames or user IDs allowed to interact. Empty list means no access (fail-closed). |
| `polling_timeout` | duration | `10s` | Long-polling timeout for updates |
| `webhook_url` | string | `""` | Public URL Telegram sends updates to. If set, enables webhook mode. |
| `webhook_listen` | string | `":8443"` | Local address for the webhook listener |
| `webhook_cert_file` | string | `""` | Path to TLS cert PEM (required for self-signed certs) |
| `webhook_key_file` | string | `""` | Path to TLS key PEM |
| `webhook_secret` | string | `""` | Secret token for request verification (Bot API 6.1+) |

```yaml
name: "telegram"
enabled: false
token_env: "GOGOCLAW_TELEGRAM_TOKEN"
# The bot will not respond until at least one username or user ID is added below.
allowed_users: []
polling_timeout: 10s
# Webhook mode (optional — leave webhook_url empty for long-polling):
# webhook_url: "https://your-domain.com/telegram/webhook"
# webhook_listen: ":8443"
# webhook_cert_file: ""
# webhook_key_file: ""
# webhook_secret: ""
```

**Security note:** The `allowed_users` list is fail-closed. An empty list means no one can interact with the bot. You must explicitly add usernames or Telegram user IDs before the bot will respond to anyone.

**Webhook vs. long-polling:** Long-polling (default) is simpler to set up and works behind NAT, making it ideal for development. Webhook mode provides lower latency and better resource efficiency but requires a public HTTPS URL reachable by Telegram's servers.

## Network Config (network.yaml)

The network allowlist controls all outbound HTTP from GoGoClaw. All traffic from providers, the `web_fetch` tool, and MCP SSE connections goes through this allowlist.

```yaml
allowlist:
  - "localhost"
  - "127.0.0.1"
  - "api.openai.com"
deny_all_others: true
log_blocked: true
```

### Fields

| Field | Type | Default | Description |
|---|---|---|---|
| `allowlist` | list | `["localhost", "127.0.0.1"]` | Domains permitted for outbound HTTP |
| `deny_all_others` | bool | `true` | Block all domains not in the allowlist |
| `log_blocked` | bool | `true` | Log blocked connection attempts |

The bootstrap process auto-generates this file with `localhost`, `127.0.0.1`, and the domains extracted from your configured provider base URLs. If you add new providers or MCP SSE servers later, you must add their domains to this list manually.

## Memory Config (memory/config.yaml)

The memory system provides vector-backed long-term memory using embeddings. Configuration can be placed in `~/.gogoclaw/memory/config.yaml` (takes precedence) or inline under the `memory` key in `config.yaml`.

```yaml
enabled: true

embedding:
  provider: "openai"
  model: "text-embedding-3-small"
  fallback_provider: "ollama"

storage:
  backend: "chromem"
  path: "~/.gogoclaw/memory/vectors"
  encrypted: false

retrieval:
  top_k: 10
  relevance_threshold: 0.7
  recency_weight: 0.2
```

### Fields

| Field | Type | Default | Description |
|---|---|---|---|
| `enabled` | bool | `false` | Enable the vector memory system |
| `embedding.provider` | string | -- | Provider name for generating embeddings |
| `embedding.model` | string | -- | Embedding model identifier |
| `embedding.fallback_provider` | string | -- | Fallback provider if primary is unavailable |
| `storage.backend` | string | -- | Storage backend (e.g. `chromem`) |
| `storage.path` | string | -- | Path for vector storage files |
| `storage.encrypted` | bool | `false` | Encrypt stored vectors |
| `retrieval.top_k` | int | -- | Number of results to retrieve per query |
| `retrieval.relevance_threshold` | float | -- | Minimum cosine similarity to include a result |
| `retrieval.recency_weight` | float | -- | Weight for recency vs relevance scoring (0.0-1.0) |

## PII Modes

The PII gate scans all non-assistant messages (system, user, and tool messages) before they are sent to the LLM provider.

### Modes

| Mode | Behavior |
|---|---|
| `strict` | Block requests containing PII from reaching cloud providers. Requests to local providers (e.g. Ollama) are allowed. |
| `warn` | Flag detected PII and notify the user, but proceed with the request. |
| `permissive` | Log PII detections to the audit log only. No user notification or blocking. |
| `disabled` | No PII detection or scanning. |

### Detected patterns

The PII classifier detects the following patterns:

- **SSN** -- Social Security Numbers in `XXX-XX-XXXX` or 9-digit format (with area/group/serial validation)
- **Phone numbers** -- US phone numbers in various formats
- **Email addresses** -- Standard email address patterns
- **Credit card numbers** -- 16-digit card numbers with Luhn algorithm validation
- **Account numbers** -- Patterns matching `acct#123456` and similar formats
- **API keys** -- Patterns starting with `sk-`, `ghp_`, or `AKIA` (AWS access keys)
- **Bearer tokens** -- `Bearer` followed by a token string

### Per-conversation overrides

PII mode can be overridden on a per-conversation basis. This allows temporarily relaxing or tightening PII controls without changing the agent config. Overrides are set programmatically and do not persist across restarts.

### Configuration

PII mode is set in the agent config:

```yaml
pii:
  mode: "strict"   # strict | warn | permissive | disabled
```

## MCP Server Configs (mcp/*.yaml)

Each file in `~/.gogoclaw/mcp/` defines an MCP (Model Context Protocol) server connection. Tools exposed by MCP servers are namespaced as `mcp_{servername}_{toolname}` to avoid collisions with built-in tools.

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | No | Display name (defaults to filename) |
| `transport` | string | Yes | `stdio` or `sse` |
| `command` | string | For stdio | Command to execute for stdio transport |
| `args` | list | No | Arguments to pass to the stdio command |
| `url` | string | For sse | URL for SSE transport |
| `enabled` | bool | Yes | Whether the server is active |

### Example: stdio transport (mcp/filesystem.yaml)

```yaml
name: "filesystem"
transport: "stdio"
command: "npx"
args:
  - "-y"
  - "@modelcontextprotocol/server-filesystem"
  - "/home/user/documents"
enabled: true
```

### Example: SSE transport (mcp/remote-tools.yaml)

```yaml
name: "remote-tools"
transport: "sse"
url: "https://mcp.example.com/sse"
enabled: true
```

**Important:** For SSE transport servers, the server domain must be added to the network allowlist in `network.yaml`. For the example above, you would add `mcp.example.com` to the `allowlist`.

## Environment Variables

GoGoClaw loads environment variables from `~/.gogoclaw/env` at process start, before config files are read. This allows `${VAR_NAME}` references in YAML configs to resolve correctly.

### File format

The env file uses a simple `KEY=value` format, one entry per line. Lines starting with `#` are comments. Blank lines are ignored.

```
# LLM provider keys
OPENAI_API_KEY=sk-proj-abc123...
GROQ_API_KEY=gsk_abc123...

# Channel tokens
GOGOCLAW_TELEGRAM_TOKEN=123456:ABC-DEF...
GOGOCLAW_REST_API_KEY=your-rest-api-key
```

### Precedence

System environment variables take precedence over the env file. If a variable is already set in the system environment, the value from `~/.gogoclaw/env` is skipped and a log message is emitted. This means you can override any env file value by setting the variable in your shell or system environment.

### Security

The env file is written with `0600` permissions (owner read/write only). On Windows, POSIX permissions are not enforced -- the file relies on the user home directory inherited NTFS ACLs for protection, which by default restrict access to the owning user, Administrators, and SYSTEM.

Secret values entered during bootstrap are collected directly (with hidden input when running in a terminal) and written to this file without passing through the LLM engine, audit logger, or any persistence layer.

On Windows, the bootstrap process also calls `setx` to persist environment variables at the system level for availability in future sessions.
